# Go 内存管理与垃圾回收

---

## 一、Go 的内存分配

### 面试问题

Go 的内存分配机制是什么？栈和堆有什么区别？如何判断变量逃逸？

### 标准答案

**栈 vs 堆：**

```go
// 栈（Stack）
// - 每个 goroutine 有独立的栈（初始 2KB，动态增长）
// - 分配和回收极快（移动栈指针）
// - 存储局部变量
// - 函数返回时自动释放

// 堆（Heap）
// - 全局共享
// - 分配和回收需要 GC
// - 存储逃逸的变量
// - 生命周期不确定
```

**逃逸分析（Escape Analysis）：**

```go
// 编译器决定变量在栈上还是堆上分配
// 如果变量逃逸到函数外部，则分配在堆上

func foo() *int {
    x := 42
    return &x  // x 逃逸到堆上（返回了指针）
}

func bar() int {
    x := 42
    return x  // x 在栈上（值拷贝）
}

// 查看逃逸分析结果
// go build -gcflags="-m" main.go
// go build -gcflags="-m -m" main.go  (更详细)
```

**常见的逃逸场景：**

```go
// 1. 返回局部变量的指针
func create() *int {
    x := 42
    return &x  // 逃逸
}

// 2. 发送到 channel
func send(ch chan<- int) {
    x := 42
    ch <- x  // x 可能逃逸
}

// 3. 赋值给接口变量
func process(i interface{}) {
    // ...
}

func main() {
    x := 42
    process(x)  // x 逃逸（装箱）
}

// 4. 闭包引用
func counter() func() int {
    count := 0  // 逃逸（被闭包引用）
    return func() int {
        count++
        return count
    }
}

// 5. slice 和 map 的元素可能逃逸
func process() {
    s := make([]int, 100)
    for i := range s {
        s[i] = i  // s 本身可能逃逸
    }
}

// 6. 大对象（超过一定大小）
func process() {
    var buf [1024 * 1024]byte  // 可能分配在堆上
}
```

**Go 的内存分配器（TCMalloc）：**

```go
// Go 使用 TCMalloc（Thread-Caching Malloc）的变体
// 分配流程：
// 1. 微对象（< 16B）：使用 mcache 的 tiny 分配器
// 2. 小对象（16B ~ 32KB）：使用 mcache 的 span
// 3. 大对象（> 32KB）：直接从 mheap 分配

// 内存层次结构：
// mcache (per-P) → mcentral (per-size-class) → mheap (global)
```

## 二、Go 的垃圾回收（GC）

### 面试问题

Go 的 GC 是什么算法？GC 的流程是什么？如何减少 GC 压力？

### 标准答案

**GC 算法：三色标记清除 + 写屏障**

```go
// 三色标记法：
// - 白色：未被标记的对象（潜在垃圾）
// - 灰色：已标记但子对象未处理
// - 黑色：已标记且子对象已处理

// GC 流程：
// 1. 标记准备（STW - Stop The World）
//    - 开启写屏障
//    - 扫描栈上的根对象
//
// 2. 并发标记
//    - 从根对象出发，标记所有可达对象
//    - 三色：白→灰→黑
//    - 用户代码并发执行
//
// 3. 标记终止（STW）
//    - 关闭写屏障
//    - 重新扫描栈（可能有新引用）
//
// 4. 并发清除
//    - 回收白色对象
//    - 用户代码并发执行
```

**GC 触发条件：**

```go
// 1. 内存增长到阈值（GOGC）
// 默认 GOGC=100，表示堆内存增长 100% 时触发 GC
// 上次 GC 后存活内存 10MB → 增长到 20MB 时触发

// 2. 超过 2 分钟没有 GC

// 3. 手动触发
runtime.GC()

// 设置 GOGC
GOGC=200  // 堆内存增长 200% 时触发（减少 GC 频率）
GOGC=50   // 堆内存增长 50% 时触发（增加 GC 频率）
GOGC=off  // 关闭 GC（不推荐）

// Go 1.19+ 引入了 GOMEMLIMIT
GOMEMLIMIT=1GiB  // 设置内存上限，GC 会在接近上限时更积极地回收
```

**GC 调优：**

```go
// 1. 减少内存分配
// 预分配 slice 容量
s := make([]int, 0, 1000)  // 预分配
for i := 0; i < 1000; i++ {
    s = append(s, i)  // 不会多次扩容
}

// 2. 复用对象
var pool = sync.Pool{
    New: func() interface{} {
        return new(bytes.Buffer)
    },
}

// 3. 使用值类型而非指针
type User struct {
    Name string
    Age  int
}
// 比 *User 更少的堆分配

// 4. 避免不必要的 interface{}
func process(v interface{}) {  // v 会逃逸到堆
    // ...
}

// 5. 使用 arena（Go 1.20 实验性）
// 批量分配和释放内存
```

**GC 监控和调试：**

```go
import "runtime/debug"

// 设置 GC 目标
debug.SetGCPercent(100)  // 等价于 GOGC=100

// 设置内存上限（Go 1.19+）
debug.SetMemoryLimit(1 << 30)  // 1GB

// 获取 GC 统计
var stats runtime.MemStats
runtime.ReadMemStats(&stats)
fmt.Printf("GC 次数: %d\n", stats.NumGC)
fmt.Printf("上次 GC 暂停: %v\n", time.Duration(stats.PauseTotalNs))
fmt.Printf("堆内存: %d MB\n", stats.HeapAlloc/1024/1024)

// GC trace
// GODEBUG=gctrace=1 go run main.go
// 输出：gc 1 @0.012s 2%: 0.017+1.2+0.018 ms clock, 0.068+0.35/0.62/0.14+0.072 ms cpu, 4->5->2 MB, 5 MB goal, 0 MB stacks, 0 MB globals, 4 P
```

## 三、Go 的 pprof 性能分析

### 面试问题

如何使用 pprof 分析 Go 程序的性能？有哪些类型的 profile？

### 标准答案

**在代码中启用 pprof：**

```go
import (
    "net/http"
    _ "net/http/pprof"
)

func main() {
    // 方式一：独立 HTTP 服务器
    go func() {
        log.Println(http.ListenAndServe("localhost:6060", nil))
    }()

    // 方式二：注册到现有 mux
    mux := http.NewServeMux()
    mux.HandleFunc("/debug/pprof/", pprof.Index)
    mux.HandleFunc("/debug/pprof/cmdline", pprof.Cmdline)
    mux.HandleFunc("/debug/pprof/profile", pprof.Profile)
    mux.HandleFunc("/debug/pprof/symbol", pprof.Symbol)
    mux.HandleFunc("/debug/pprof/trace", pprof.Trace)
}
```

**使用 pprof 工具：**

```bash
# CPU Profile
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30
# 交互模式：
# (pprof) top          # 查看最耗时的函数
# (pprof) top20        # 查看前 20 个
# (pprof) list funcName  # 查看函数的逐行耗时
# (pprof) web          # 生成调用图（需要 graphviz）

# Memory Profile
go tool pprof http://localhost:6060/debug/pprof/heap
# (pprof) top
# (pprof) list funcName

# Goroutine Profile
go tool pprof http://localhost:6060/debug/pprof/goroutine
# 查看所有 goroutine 的堆栈

# Block Profile（阻塞分析）
go tool pprof http://localhost:6060/debug/pprof/block

# Mutex Profile（锁竞争分析）
go tool pprof http://localhost:6060/debug/pprof/mutex

# Trace（执行追踪）
curl -o trace.out http://localhost:6060/debug/pprof/trace?seconds=5
go tool trace trace.out
# Web 界面，可以看到 goroutine 的调度、GC 事件、系统调用等

# Web 界面
go tool pprof -http=:8080 http://localhost:6060/debug/pprof/profile?seconds=30
```

**在测试中使用 pprof：**

```go
import "testing"

func BenchmarkSomething(b *testing.B) {
    for i := 0; i < b.N; i++ {
        doSomething()
    }
}

// 运行基准测试并生成 profile
// go test -bench=. -cpuprofile=cpu.prof -memprofile=mem.prof
// go tool pprof cpu.prof
// go tool pprof mem.prof
```

**常见性能问题：**

```go
// 1. CPU 热点
// 通过 CPU profile 找到最耗时的函数
// 优化：减少计算、使用更快的算法

// 2. 内存泄漏
// 通过 heap profile 找到分配最多的函数
// 检查：goroutine 泄漏、未关闭的资源、全局缓存

// 3. Goroutine 泄漏
// 通过 goroutine profile 查看 goroutine 数量
// 检查：未关闭的 channel、阻塞的 HTTP 请求

// 4. 锁竞争
// 通过 mutex profile 找到锁等待时间最长的地方
// 优化：减少锁的粒度、使用无锁数据结构

// 5. 阻塞操作
// 通过 block profile 找到阻塞最久的操作
// 优化：使用异步 IO、减少锁等待
```

## 四、Go 的内存对齐

### 面试问题

什么是内存对齐？如何优化结构体的内存占用？

### 标准答案

**内存对齐规则：**

```go
// Go 的内存对齐规则：
// - bool, int8, uint8: 1 字节
// - int16, uint16: 2 字节
// - int32, uint32, float32: 4 字节
// - int64, uint64, float64, pointer: 8 字节
// - struct: 最大字段的对齐
// - array: 元素类型的对齐

// 结构体字段按声明顺序排列，每个字段的偏移量必须是对齐值的倍数

// 不好的顺序（浪费内存）
type Bad struct {
    a bool    // 1 byte + 7 padding = 8 bytes
    b int64   // 8 bytes
    c bool    // 1 byte + 3 padding = 4 bytes
    d int32   // 4 bytes
}
// 总共：24 bytes

// 好的顺序（节省内存）
type Good struct {
    b int64   // 8 bytes
    d int32   // 4 bytes
    a bool    // 1 byte
    c bool    // 1 byte + 2 padding = 4 bytes
}
// 总共：16 bytes

// 使用 fieldalignment 工具检查
// go install golang.org/x/tools/go/analysis/passes/fieldalignment/cmd/fieldalignment@latest
// fieldalignment -fix ./...
```

**面试加分点：** 内存对齐对性能有重要影响。未对齐的内存访问可能需要多次内存读取，而对齐的访问只需要一次。在设计结构体时，应该按字段大小从大到小排列。
