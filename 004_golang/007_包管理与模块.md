# Go 包管理与模块

## 一、Go 的包（package）

### 面试问题

Go 的包是什么？导出规则是什么？init 函数的执行顺序是什么？

### 标准答案

**包的基本规则：**

```go
// 1. 每个 Go 源文件必须属于一个包
package main

// 2. 同一个目录下的所有 .go 文件必须属于同一个包

// 3. 包名通常与目录名一致（但不是必须）

// 4. main 包是程序入口，必须有 main() 函数
```

**导出规则：**

```go
// 首字母大写：导出（public）
// 首字母小写：未导出（private）

package math

// 导出
type Point struct {
    X float64  // 导出
    Y float64  // 导出
}

func Distance(p1, p2 Point) float64 {  // 导出
    // ...
}

// 未导出
type vector struct {  // 未导出
    x float64
    y float64
}

func dotProduct(v1, v2 vector) float64 {  // 未导出
    // ...
}

// 外部包只能访问导出的标识符
import "math"

p := math.Point{X: 1, Y: 2}  // OK
math.Distance(p1, p2)        // OK
// math.vector{}              // 编译错误：未导出
```

**import 规则：**

```go
// 标准库
import (
    "fmt"
    "os"
    "net/http"
)

// 第三方库
import (
    "github.com/gin-gonic/gin"
    "github.com/go-redis/redis/v9"
)

// 本地包
import (
    "myproject/pkg/utils"
    "myproject/internal/handler"
)

// 别名
import (
    myfmt "fmt"
    . "fmt"      // 导入所有导出的标识符（不推荐）
    _ "github.com/go-sql-driver/mysql"  // 匿名导入（触发 init）
)

// go:embed 指令（Go 1.16+）
import _ "embed"

//go:embed config.json
var configData []byte

//go:embed templates/*.html
var templates embed.FS
```

**init 函数的执行顺序：**

```go
// 执行顺序：
// 1. 依赖包的 init
// 2. 当前包的 init
// 3. main 包的 init
// 4. main 函数

// 依赖图：
// main → pkgA → pkgB
//       → pkgC
// 执行顺序：pkgB.init → pkgC.init → pkgA.init → main.init → main.main
```

## 二、Go Modules

### 面试问题

Go Modules 是什么？如何管理依赖？go.mod 和 go.sum 有什么作用？

### 标准答案

**基本命令：**

```bash
# 初始化模块
go mod init github.com/username/project

# 添加依赖
go get github.com/gin-gonic/gin
go get github.com/gin-gonic/gin@v1.9.0
go get github.com/gin-gonic/gin@latest

# 更新依赖
go get -u github.com/gin-gonic/gin
go get -u ./...  # 更新所有依赖

# 移除未使用的依赖
go mod tidy

# 下载依赖到本地缓存
go mod download

# 查看依赖图
go mod graph

# 查看为什么需要某个依赖
go mod why github.com/some/package

# 编辑 go.mod
go mod edit -require github.com/gin-gonic/gin@v1.9.0
go mod edit -replace github.com/old/pkg=github.com/new/pkg@v1.0.0

# vendor 模式
go mod vendor  # 将依赖复制到 vendor 目录
go build -mod=vendor  # 使用 vendor 目录构建
```

**go.mod 文件：**

```go
module github.com/username/project

go 1.21

require (
    github.com/gin-gonic/gin v1.9.0
    github.com/go-redis/redis/v9 v9.0.0
    gorm.io/gorm v1.25.0
)

// 排除特定版本
exclude github.com/some/pkg v1.2.3

// 替换依赖（本地开发常用）
replace github.com/old/pkg => ../local/pkg
replace github.com/old/pkg => github.com/new/pkg v1.0.0

// 撤回特定版本
retract v1.0.0
retract [v1.0.0, v1.0.1]
```

**go.sum 文件：**

```bash
# go.sum 包含依赖的哈希校验值
# 用于验证依赖的完整性
github.com/gin-gonic/gin v1.9.0 h1:abc123...
github.com/gin-gonic/gin v1.9.0/go.mod h1:def456...

# go.sum 应该提交到版本控制
```

**依赖版本语义：**

```bash
# 语义化版本：v主版本.次版本.修订版本
v1.2.3

# v0.x.x：不稳定版本
v0.1.0

# v1.x.x：稳定版本
v1.0.0

# v2+：主版本号变更，模块路径必须包含版本号
module github.com/user/pkg/v2

import "github.com/user/pkg/v2"
```

## 三、Go 的项目结构

### 面试问题

Go 项目应该如何组织目录结构？有哪些最佳实践？

### 标准答案

**标准布局（Standard Layout）：**

```
project/
├── cmd/                    # 可执行文件入口
│   ├── server/
│   │   └── main.go
│   └── cli/
│       └── main.go
├── internal/               # 私有代码（不能被外部导入）
│   ├── handler/            # HTTP 处理器
│   ├── service/            # 业务逻辑
│   ├── repository/         # 数据访问
│   └── model/              # 数据模型
├── pkg/                    # 可以被外部导入的公共库
│   ├── utils/
│   └── middleware/
├── api/                    # API 定义（protobuf, OpenAPI）
├── configs/                # 配置文件
├── scripts/                # 脚本
├── deployments/            # 部署配置
├── docs/                   # 文档
├── test/                   # 测试数据和工具
├── go.mod
├── go.sum
├── Makefile
└── README.md
```

**目录职责：**

```go
// cmd/：程序入口
// 每个子目录对应一个可执行文件
// cmd/server/main.go
package main

import (
    "myproject/internal/handler"
    "myproject/internal/service"
)

func main() {
    svc := service.NewUserService(repo)
    h := handler.NewHandler(svc)
    h.Run(":8080")
}

// internal/：私有代码
// Go 编译器强制 internal 包只能被其父目录导入
// internal/handler/user.go
package handler

// pkg/：公共库
// 可以被其他项目导入
// pkg/utils/validator.go
package utils

// 外部导入
import "github.com/user/project/pkg/utils"
```

**依赖注入：**

```go
// 定义接口
type UserRepository interface {
    FindByID(id int) (*User, error)
    Save(user *User) error
}

// 实现接口
type postgresUserRepo struct {
    db *sql.DB
}

func NewPostgresUserRepo(db *sql.DB) UserRepository {
    return &postgresUserRepo{db: db}
}

// 业务层依赖接口
type UserService struct {
    repo UserRepository
}

func NewUserService(repo UserRepository) *UserService {
    return &UserService{repo: repo}
}

// main.go 中组装
func main() {
    db := connectDB()
    repo := NewPostgresUserRepo(db)
    svc := NewUserService(repo)
    handler := NewHandler(svc)
    handler.Run(":8080")
}
```

**面试加分点：** `internal` 目录是 Go 编译器强制的包可见性约束。任何包如果其路径包含 `internal`，则只能被 `internal` 的父目录树中的代码导入。

## 四、Go 的构建和工具链

### 面试问题

Go 有哪些常用的工具命令？如何进行交叉编译？

### 标准答案

**常用命令：**

```bash
# 构建
go build ./...              # 构建所有包
go build -o server cmd/server/main.go  # 指定输出文件
go build -ldflags "-s -w"   # 去除调试信息，减小体积

# 运行
go run main.go              # 编译并运行
go run .                    # 运行当前目录

# 测试
go test ./...               # 运行所有测试
go test -v                  # 详细输出
go test -run TestXxx        # 运行特定测试
go test -bench .            # 运行基准测试
go test -cover              # 测试覆盖率
go test -coverprofile=coverage.out
go tool cover -html=coverage.out  # 生成 HTML 报告
go test -race               # 竞态检测

# 格式化
go fmt ./...                # 格式化代码
goimports -w .              # 格式化 import（需安装）

# 静态分析
go vet ./...                # 静态检查
golangci-lint run           # 更全面的检查（需安装）

# 文档
go doc fmt.Println          # 查看文档
godoc -http=:6060           # 启动文档服务器

# 依赖管理
go mod init                 # 初始化模块
go mod tidy                 # 清理依赖
go mod download             # 下载依赖
go mod graph                # 查看依赖图
go mod why <package>        # 为什么需要这个依赖

# 性能分析
go test -cpuprofile=cpu.out
go test -memprofile=mem.out
go tool pprof cpu.out
go tool pprof -http=:8080 cpu.out  # Web 界面

# 代码生成
go generate ./...           # 运行代码生成器
```

**交叉编译：**

```bash
# Go 原生支持交叉编译
# 设置 GOOS 和 GOARCH 环境变量

# Linux
GOOS=linux GOARCH=amd64 go build -o server-linux

# Windows
GOOS=windows GOARCH=amd64 go build -o server.exe

# macOS
GOOS=darwin GOARCH=amd64 go build -o server-mac

# ARM（如树莓派）
GOOS=linux GOARCH=arm go build -o server-arm

# 查看支持的平台
go tool dist list

# 静态编译（适合 Docker）
CGO_ENABLED=0 GOOS=linux go build -a -ldflags '-extldflags "-static"' -o server
```

**构建标签（Build Tags）：**

```go
// +build linux darwin
//go:build linux || darwin

// 文件只在 Linux 或 macOS 上编译

// +build !windows
//go:build !windows

// 文件不在 Windows 上编译

// 使用
go build -tags "integration" ./...

// 测试标签
//go:build integration

func TestIntegration(t *testing.T) {
    // 集成测试
}
go test -tags integration ./...
```

**go generate：**

```go
//go:generate mockgen -source=user.go -destination=mock_user.go -package=handler

//go:generate protoc --go_out=. *.proto

//go:generate stringer -type=Color

// 运行所有 generate
go generate ./...
```
